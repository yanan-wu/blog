## 功能定位
**Skill 是知识外化机制,本质上是一个知识包**

传统 AI 的知识锁在模型参数里，想让模型学会新技能只能使用训练这条成本很高的方法。 

而 Skill 就像给 base model 外挂了一个可热插拔的 LoRA 适配器，但完全不需要训练。你用自然语言编辑一个 Markdown 文件，模型的行为就改变了。
这是从"训练 AI"到"教育 AI"的范式转变。

> 通用 Agent + 优秀 Skill = 特定领域的专家 Agent

## 结构
一个 Skill 应该拆成四个部分：

```
skill-name/
├── SKILL.md          # 核心：指导 Agent 如何完成任务的知识
├── scripts/          # 可选：可执行脚本
├── references/       # 可选：参考文档
└── examples/         # 可选：示例模板、图片等资源
```

### SKILL.md
这是唯一必需的文件，也是整个技能的大脑。
它由两部分组成：头部是元数据，用特定格式标记技能的名称和描述。正文是具体指令，告诉 AI 拿到这个任务后该怎么一步步执行。

名称是技能的唯一标识。
**描述是最关键的部分，AI 通过这段话来判断什么时候该激活这个技能。写得越具体越好，要说清楚这个技能能做什么、什么场景下该用。**

建议 SKILL.md 控制在 500 行以内。超过这个长度，就应该把详细内容拆分到其他文件里。

### reference
这里放那些详细的参考资料。比如完整的 API 文档、代码规范的细节、边缘情况的处理方式。
这些内容如果都塞进 SKILL.md，每次激活都会占用大量 token。分离出来之后，AI 只在需要深入了解的时候才会去读。

### examples
这里放具体的示例。

与其用文字描述"组件应该长什么样"，不如直接放一个写好的组件让 AI 参考。AI 看一眼就知道格式该是什么样，比文字描述有效得多。

### scripts 
这里放可执行的脚本。

有些事情你教 AI 十遍它也可能出错，比如"检查代码格式是否符合规范"。与其用文字描述规则让它判断，不如写一个脚本直接执行检查。

AI 运行脚本，只看输出结果。逻辑是固定的，不会出现理解偏差。 而且执行脚本不需要读源码，零 token 成本。

## 加载机制
Skill 设计的核心理念是 按需加载上下文 ——把详细知识存在 reference 文件里，需要时才加载，保持 context window 精简。

加载过程分三层：发现 -> 激活 -> 按需读取
- 第一层是发现。 AI 启动时会扫描所有 Skill，但只读取每个 SKILL.md 的头部元数据，也就是名称和描述。 这样它知道有哪些技能可用，但不会占用太多上下文。
- 第二层是激活。 当用户的请求和某个技能的描述匹配时，AI 才会加载这个 SKILL.md 的完整内容。 这时候它拿到了具体指令，知道该怎么做，也知道有哪些资源可以用。
- 第三层是按需读取。执行过程中，AI 看到 SKILL.md 里提到的那些引用，会根据需要决定是否去读取。 需要看规范就读 reference下的文件，需要看示例就读 examples下的文件，需要验证就执行 scripts 里的脚本。

不需要的文件，始终不会被读取。这就是渐进式加载。省 token，也让 AI 更专注。
